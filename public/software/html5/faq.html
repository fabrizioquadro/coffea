6. SNI (server name indication) 
Html5 client has to be always first instance in network chain, it plays the role of SSL decrypter. 
Even if you decide to use third web servers like IIS/Apache, the html5 instance has to be still the SSL decrypter,
else no websocket connectivity will be available. The html5 instance has to decrypt it to distinguish packets to be correctly forwarded.
Let's assume you have 3 different domains pointing to same IP
my-domain.com (200.200.200.200 -> 127.0.0.1)
second.my-domain.com (200.200.200.200 -> 127.0.0.1)
some-domain.com (200.200.200.200 -> 127.0.0.1)

In order to serve different certificates according to domain name do following.

1. create subfolder "multicerts" under "webserver" folder.

2. rename *.jks files for example above
https://my-domain.com/ > ***\webserver\multicerts\my-domain.com.jks
https://second.my-domain.com/ > ***\webserver\multicerts\second.my-domain.com.jks
https://some-domain.com/ > ***\webserver\multicerts\some-domain.com.jks

(if you have multidomain certificate *.my-multidomain.com and want to serve this certificate
for all domains ending with .my-multidomain.com inclusive my-multidomain.com then add leading point to jks certificates name
https://****.my-multidomain.com/ > ***\webserver\multicerts\.my-multidomain.com.jks)

3. restart html5 in GUI or reread configuration http://127.0.0.1/w0j3?cert.jks

In such case when ever you access
https:\\my-domain.com, https:\\second.my-domain.com or https:\\some-domain.com you will get different certificates served 
(however this requires SNI support by browser, but all modern browsers support it for longer time)
For other unrecognized (not in "multicerts" folder) domains the default \webserver\cert.jks will be served
This SNI feature makes sense only for authority signed certificates, for self signed certificates one such cert is enough, 
since it makes no sense to serve different self signed certs.

21. Mutual auth on SSL 
To understand what that is in comparison to normal SSL is the extra check of browser if browser is authorized to create SSL connection.
The default browser behaviour by accessing https://server.com server is to check if server is autorized instance for browser, and by mutual auth additionally to previous step 
the server self after being autorized by  browser plays the role of browser and checks back if browser is autorized too. Also they both check them vice versa for being authorized.
That mutual auth step requests installation of matching keypair into browsers repository so that browser can be autorized by server.
To make it all working
1. open/edit(create) \webserver\settings.bin 

2. add following
force_mutual_auth_on_https=true
disable_http_only=true

3. copy actual \webserver\cert.jks somewhere else just for case of backup

4. create batch file in same folder with \webserver\cert.jks, as example \webserver\createcertuser.bat with following content, at least for test
@rem uncomment next line, if you want to generate new self signed cert.jks
@rem keytool -genkey -v -alias jwts -keyalg RSA -validity 3650 -keystore cert.jks -storepass secret -keypass secret -dname "CN=localhost, OU=my_ou, O=my_org, L=my_city, ST=my_state, C=MY"
@keytool -genkey -v -alias AliasUser1 -keyalg RSA -storetype PKCS12 -keystore forBrowserUser1.p12 -dname "CN=Mr. Wagner, OU=some_ou, O=some_org, L=Paris, ST=UK, C=FR" -storepass mypassword -keypass mypassword
@keytool -export -alias AliasUser1 -keystore forBrowserUser1.p12 -storetype PKCS12 -storepass mypassword -rfc -file forCertUser1.cer
@keytool -alias AliasCertUser2 -import -v -file forCertUser1.cer -keystore cert.jks -storepass secret
@del forCertUser1.cer

It will also import automatically the cert of browsers keypair into cert.jks after creating this browsers keypair
(but not browsers keypair self, it will not be imported into cert.jks, since not necessary to be imported on server, but only on browser)
keytool.exe is part of Java JDK, so possibly you will need to install Java JDK firstly to be able to use this tool

5. the batch example should have generated test file forBrowserUser1.p12 in step 3.
If you open in browser the https page for first time, you will get security message and you will be unable to access webserver pages.. 
For Chrome and IE it is enough to click on this forBrowserUser1.p12 file and import into Windows default keystore, as soon you imported it, you will be able to access https address.
By FireFox you have to go to settings and import this file under your certificates
about:preferences#advanced
only so you will be able to open it on FireFox.

26. HTML5 full screen. 
In order to support full screen mode the server has to support either on-the-fly-resizing or reconnection-on-resize or be prepared for fullscreen mode.
On-the-fly-resizing requests support of dynamic channels with Display channel that is available in newer Windows systems like Windows2012R2, Windows10 etc.

Server supports on-the-fly-resizing (dynamic Display channel)

1. no future configuration needed, the full screen button will be present in top menu, if not forcibly disabled (*\settings.js -> W.showfullscreenbutton = "no")

2. even if server supports on-the-fly-resizing you still may configure/edit www\software\html5\settings.js -> W.full_screen = 2;
   in this case the session will take from start the fully available screen size and will not anymore change it self during resizing.

Server doesn't support on-the-fly-resizing (old systems like Windows2008, Windows XP)

1. either you must configure/edit www\software\html5\settings.js -> W.reconnectonresize = true;
   in this reconnect-on-resize mode the client will try to reconnect session with new dimensions, but highly not recommended
   because if you run file transfer or sound gets played these actions will be interrupted.

2. or you must configure/edit www\software\html5\settings.js -> W.full_screen = 2;
   in this case the session will take from start the fully available screen size.

Notice: if no one of these points above (W.reconnectonresize = true or W.full_screen = 2) is effective the full screen button will be deactivated. 

Smartfit setting, only effective if server doesn't support on-the-fly-resizing or was preset to fullscreen (W.full_screen = 2)

1. by deafult www\software\html5\settings.js -> W.smartfit = true; is enabled. (* = 1)
   in this case if rdp area overflows browser area the rdp area will be squeezed to fit browser area.

2. or configure/edit www\software\html5\settings.js -> W.smartfit = false; to disable it. (* = 0)
   in this case if rdp area overflows browser area then the browser tab will automatically display scroll bars.

3. smartfit set to true or 1 squeezes the rdp area by underflowing, if you set it to 2 additionally it will stretch by overflowing.

Full screen enforced with W.full_screen = 2; or W.full_screen = 1; 

1. by default when using enforced full screen mode any click inside working area will switch automatically to full screen.
   to disable it edit settings.js > W.allowOnclickFullScreen = false;

31. Java VM memory on server to low: ***. 
If you get this error by starting html5 client according to your situation you get proposal either to install Java-64-bit or extend your RAM memory.
Remember, any 32bit application can not access more than 4Gb RAM, but Java computes its maximal usable RAM amount so: Actual-Free-Physical-Memory / 4,
also 25% of free physical memory. That means if you have 4Gb RAM / 4 = 1Gb only. But mostly the amount of free memory usable for 32bit applications
is even lower. You can check it by calling link http://localhost/w0j4 but according to experience on Windows-64-bit with Java-32-bit this amount is mostly about ~250 Mb.
It makes sense to install Java-64-bit only if you have Windows-64-bit!

1. open link in browser https://java.com/en/download/manual.jsp
2. download Windows Offline (64-bit)
3. install Java-64-bit
4. restart html5 server via GUI to make this change effective
   (you can but you do not need to deinstall Java-32-bit since GUI will automatically prefer Java-64-bit instead its Java-32-bit version)

There are cases when server has as example 30Gb RAM installed with Windows-32-bit, in such cases only what could help you is to upgrade to Windows-64-bit because 
Windows-32-bit can not use more than 4Gb RAM and 1Gb of RAM is already used by Windows-System and so finally you have only 3Gb RAM usable with all consequences.

36. Printing in raw format. 
Important notice: PDF files can not be printed to RAW printer, RAW prints can not be converted to PDF (there is no ready to use RAW to PDF converter provided)
Finally you can not print to barcode or other RAW type printer from inside html5 rpd session. Supported are only PostScript, XPS and PCL6 print formats.
Raw format is officially unsupported by html5 client and there will be no support for it on common way since technically not doable with browsers onboard tools.
Since version html5_4.97 the program correctly recognizes if print can be converted to PDF format to avoid blank page instead print.
In case if program recognized RAW format you have two options how to handle.

A. by default without future configuration all RAW print files will be provided as downloadable files.
or
B. raw prints can be sent via registered protocol to specific program that will send the print to wished printer directly bypassing printer driver.
If you decided your self for approach B. you need to prepare not only server but all clients to be able to accept self registered protocols.
Important notice, raw printing is officially unsupported any everything what you do in following steps you do at your own risk!
On your server 
  1. locate/open/edit(create) file \webserver\settings.bin (if not found, create this file manually with Notepad)
  2. add there as next line
     unknown_print_protocol="printraw://"
  3. restart html5 in GUI to take change effect (after go sure weblog.txt contains - Set unknown print protocol: printraw://)
  this finishes server preparation
On your client where you are going to use raw direct printing via browser 
  1. download and execute http://your_server.com/software/html5/imgs/scr.zip > customprint/register_raw_protocol.reg

38. ClipBoard text usage with html5 client. 
Remember, access to the clipboard on browsers level is highly privileged security related action and may not work at all depending on system or browser you use.  

1. to paste clipboard text data to session use Ctrl + V, this works mostly on all browsers, on Mac browsers Ctrl button has no effect, there you must use it with Right_CMD + V
2. to copy text data to your local environment use Ctrl + C or for cut Ctrl + X, on Mac Ctrl has no effect, use it with Right_CMD + C or Right_CMD + X 
   (Mac Safari has bug and does not wait for data from server on privileged action or adds data only on second cmd+c pressing)
3. to copy data by mouse to your local browser environment is very tricky and requests browser support of ClipBoard API. On each mouse click you give the system grace time of 1 second, if in this time
the data arrives from server and ClipBoard Api is supported, this data will be added to your system, but if the data was to late because of slow speed between server or data amount was just to big, then this
data will be added to server on next(second) click if ClipBoard Api supported.
4. there is no technical way to paste clipboard data by mouse click from your local environment and send it to server therefore use Ctrl + V (or on Mac Right_CMD + V). 
Chrome and FireFox considered it as big potential security risk and do not allow that even on privileged action like mouse click. However Internet Explorer partially supports it 
but with displaying of security message which when not accepted will break future clipboard input therefore this approach is not implemented for IE since considered as unstable.

PS: copy files between browser and rdp session via ClipBoard access by Ctrl + V or by mouse click is technically not possible! 
Use drag-and-drop instead to send to session and send files back to browser via \\tsclient\WebServer
39. ClipBoard image usage with html5 client. 
By image copy/pasting you get same limitations like on ClipBoard text transfer from answer 38. but with even more limiations.
1. To paste image data from your computer into rdp session just press Ctrl + V (on Mac Safari has bug and does not allow to access such pasted images therefore use Mac Chrome or Mac FireFox if you need to paste images on Mac)
   (there is no way to paste images by mouse click)
2. To copy image to your local environment is tricky, you may however use Ctrl + C, but this will not add the image from inside rdp session to your local clipboard automatically.
   Ctrl + C will just initialize sending of image clipboard data to browser, with same success you may initialize image clipboard data transfer with the mouse. When image arrives on your browser you will see shortly
   small preview of copied image area on top right side of your browser, as soon you saw that image arrived indicated by preview just make next click with right mouse button, this will initialize browsers native handler
   for right click. From that menu you must choose to copy the image to your local clipboard environment. Depending on browser you will see different menu and entries of the native handler for right click.
   Actually image clipboard transfer was successfully tested on modern/newest versions of IE11, IE12(Edge), Chrome, FireFox, and on Mac browsers too. However on Mac Safari as mentioned you can not paste images!

42. Set own cert/private key pass and store pass at run time. 
By default GUI is preset to "secret" as pass for private key and for store. GUI has no default ability to change it manually,
however you may edit \webserver\runwebserver.bat and change the passwords *** secret secret *** to your own, but it requests to
set read-only attribute on runwebserver.bat file to not allow GUI to overwrite it and then restart html5 server.
But this would still reveal real password if someone gains admin access to your server. 
To mitigate that scenario you can reuse one hidden option. 

1. locate httpwebs*.jar > keymaps\src.zip > set_cert_pass.au3 
2. insert your own passes for cert(private-key) and store
3. compile it with AutoIt to set_cert_pass.exe 
4. go sure html5 client service runs
5. and finally start set_cert_pass.exe on that server
6. program will call the link http://127.0.0.1/w0j3?cert.jks#certpass=secret#storepass=secret
7. and server will accept it as new passes until next html5 server restart.

Notice: browsers will truncate everything after # so opening the link in browser won't bring desired result, use provided AutoIt script
or urlencode the string before opening in browser #certpass=secret#storepass=secret >>> %23certpass%3Dsecret%23storepass%3Dsecret

#certpass=newsecret#storepass=newsecretstore#oldcertpass=secret#oldstorepass=secret
You will need to redo the steps 4-5 each time you restart html5 server.

PS: if you want to prohibit such live change then open/edit/(create) \webserver\settings.bin and add as next line > allow_cert_pass_change=false

50. Gateway function > back-balance 
Since HTML5 version 7.82 back balance supported by HTML5 gateway, if gateway is accessible by client and subserver when gateway is unable to access
subservers directly. By back-balance feature subgateway creates the control connection to gateway, and after subserver creates
new TCP connections to gateway on request from gateway via initially created control connection. This feature requires preparation of subserver 
settings too for creating first initial control connection.

On gateway side 
1. open/edit(create) \webserver\balance.bin
2. and add/save 
/~~srvBACK=mypassword1234#:1;
or
/~~srvBACK=mypassword1234#:*;
   (password in that example > mypassword1234 has mandatory hash char # on end, this example accepts any IP)

or
/~~srvBACK=mypassword1234#:*51.254.22.85;
   (like previously but gateway accepts connections from single IP 51.254.22.85 only)

or
/~~srvBACK=mypassword1234#:*51.254.*;
   (like previously but gateway accepts connections from IP range 51.254.0.0 to 51.254.255.255)

or
/~~srvBACK=mypassword1234#:*demo.mydyndns.net;
   (like previously but gateway accepts connections from single IP resolved for demo.mydyndns.net)

or
SSL /~~srvBACK=mypassword1234#:*; 
   (like previously but connection gets encrypted additionally by SSL for HTTP requests)

3. restart html5 via GUI or reread configuration http://127.0.0.1/w0j3?settings.bin

On subserver side - mandatory preparation step!!!
1. open/edit(create) \webserver\backbalance.bin (attention, not anymore balance.bin!!!)
2. and add/save 
   /~~srvBACK=mypassword1234#demo.tsplus.net:443;
   (will connect to gateway demo.tsplus.net:443, as by example with password used > mypassword1234 and mandatory hash char # on end)

or
/~~srvBACK=mypassword1234#demo.tsplus.net:443 -> \myuser1:mypass@192.168.10.11:808;
   (this will connect to demo.tsplus.net:443 via proxy 192.168.10.11:808 by using NTLM logon: \myuser1 and password mypass)

or
/~~srvBACK=mypassword1234#demo.tsplus.net:443 -> mydomain\myuser1:mypass@192.168.10.11:808;
   (this will connect to demo.tsplus.net:443 via proxy 192.168.10.11:808 by using NTLM logon: mydomain\myuser1 and password mypass)

or
/~~srvBACK=mypassword1234#demo.tsplus.net:443 -> myuser1:mypass@192.168.10.11:808;
   (this will connect to demo.tsplus.net:443 via proxy 192.168.10.11:808 by using Basic logon: myuser1 and password mypass)

or
/~~srvBACK=mypassword1234#demo.tsplus.net:443 -> 192.168.10.11:808;
   (this will connect to demo.tsplus.net:443 via proxy 192.168.10.11:808 by using no password)

or
/~~srvBACK=mypassword1234#demo.tsplus.net:443 -> \myuser1:mypass@192.168.10.11:808 -> \myuser2:mypass@192.168.12.12:809;
   (same like before but additionally next cascade proxy server used 192.168.12.12:809)

3. restart HTML5 via GUI to take changes effect, hot refresh not supported for backbalance.bin!!!

Important, due usage of TLS in client mode on subservers ensure that subservers have up-to-date Java version, or at least equal to Java version on gateway!
